# Docs Update Map

This file defines where each change must be documented.

## Core Product Intent
- `docs/01_summary.md`
- Update when objective, fixed study constants, or outcome targets change.

- `docs/02_problem_and_users.md`
- Update when research questions, participant profile, or problem framing changes.

## Functional Behavior
- `docs/03_solution_scope.md`
- Update when screens, sequence contracts, corpus usage, or in/out scope changes.

- `docs/04_features.md`
- Update when any requirement or acceptance criterion changes.

- `docs/05_user_flow.md`
- Update when step order, validation gates, or device controls change.

## Technical Contract
- `docs/06_tech_stack_and_integrations.md`
- Update when stack, extraction/tokenization logic, persistence model, or constraints change.

- `docs/13_data_schema.md`
- Update when JSON fields, event structure, or required payload contracts change.

## Delivery / Risk Management
- `docs/07_mvp_plan.md`
- Update when build phases, done criteria, or git workflow rules change.

- `docs/08_risks_and_open_questions.md`
- Update when risks/open questions are added, resolved, or re-prioritized.

## Traceability / Copy Contract
- `docs/11_audio_requirements_trace.md`
- Update when new audio-derived requirements are added or clarified.

- `docs/12_screen_copy_and_controls.md`
- Update when screen-level text, button labels, or control mapping changes.

## Mandatory Per Change
- `docs/09_change_log.md`
- Add one short entry after every meaningful change.

## README Rule
- After each meaningful change, check if `README.md` needs correction.
- Only correct existing README content when needed.
- Do not add new README sections unless owner explicitly asks.
